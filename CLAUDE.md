## Repository Overview

This is a set of embedded projects targeting the **GD32H759IMK6** (GigaDevice Cortex-M7, double-precision FPU) development board with a 5-inch 800×480 RGB565 LCD panel. All build with **Keil MDK-ARM** (ARM Compiler 6) — there is no CLI build system, no Makefiles, no CMake.

The comprehensive architecture document is `GD32H7xx_GDemWin652/GDemWin详解.md` — **PRIORITY READ** for any emWin or LCD work.

## Build

Open the `.uvprojx` file in Keil MDK-ARM V5 to build. No CLI/CI build is configured.
- Output artifacts: `.axf` (ELF), `.hex`, `.map` go to `MDK-ARM/output/`
- Scatter file is auto-generated by Keil (references non-existent `stm32-ram.sct`, Keil uses device memory map)

## Target Hardware and Memory Layout

- **MCU**: GD32H759IMK6 (Cortex-M7, 600 MHz)
- **LCD**: 5" 800×480 RGB565 parallel, driven by TLI peripheral (like STM32 LTDC)
- **Touch**: GT1151QM capacitive, I2C interface (not yet integrated into main project)
- **Ethernet PHY**: LAN8720A (RMII, not used in main project)

Frame buffer: **External SDRAM at `0xC0000000`** (768 KB = 800×480×2). Frees DTCM for RTOS heap/tasks.

MPU: SDRAM region (0xC0000000, 4MB) must be **non-cacheable** (`TEX=1, C=0, B=0, S=0` — Normal memory, non-cacheable). This ensures TLI DMA sees all CPU writes immediately.

### EmWin Architecture

```
GUITask (FreeRTOS, priority 2)
  └─ GUI_Init()
       ├─ GUI_X_Config()      → GUI_ALLOC_AssignMemory(64KB pool) ← CRITICAL!
       ├─ GUI_X_InitOS()      → xSemaphoreCreateCounting(1, 1)
       ├─ LCD_X_Config()      → GUIDRV_LIN_16 + GUICC_M565 + VRAM at 0xC0000000
       └─ LCD_X_DisplayDriver(LCD_X_INITCONTROLLER) → no-op (HW pre-initialized)
  └─ GUI_Delay(1000) = GUI_Exec() + GUI_X_Delay(vTaskDelay)
  └─ emWin draws into frame buffer → TLI DMA reads → LCD panel displays
```

### Hardware Init Sequence (exact copy of proven TLI_LCD_5inches)

```c
BOARD_ConfigMPU()     → MPU region + Enable MPU + ICache + DCache (in ONE function)
nvic_priority_group_set()
USART_Config()
exmc_synchronous_dynamic_ram_init()  → SDRAM init
BSP_LCD_Init()       → TLI clock (PLL2) + GPIO (AF14) + timing
LCD_LayerInit()      → Layer0 RGB565 config + framebuffer_init(WHITE) + tli_enable
LED_GPIO_Config()
// THEN: xTaskCreate + vTaskStartScheduler
```

## CRITICAL: Known Integration Issues and Fixes

### 1. GUI_ALLOC_AssignMemory — REQUIRED or GUI_Init() hangs

**Symptom**: `GUI_Init()` hangs before reaching `LCD_X_Config`. No fault, no output — just dead.
**Root cause**: The precompiled `GDemWin_CM7_OS_DP.lib` has a small default memory pool. GUI_Init() runs out of memory during internal initialization.
**Fix** (in `User/gui/GUIConf.c`):
```c
static unsigned char _GUI_Memory[64 * 1024] __attribute__((aligned(32)));
void GUI_X_Config(void) {
    GUI_ALLOC_AssignMemory(_GUI_Memory, sizeof(_GUI_Memory));
}
```

### 2. bsp_lcd function conflicts with emWin

emWin's built-in API (`LCD_Init`, `LCD_DrawPixel`, `LCD_GetXSize`, `LCD_GetYSize`, etc.) conflicts with `bsp_lcd.c`'s bare-metal drawing functions.
**Fix**: Both `bsp_lcd.c` and `bsp_lcd.h` use `#ifndef USE_GDEMWIN` guards to exclude conflicting drawing functions when emWin is used. Only HW init functions (`BSP_LCD_Init`, `LCD_GPIO_Config`, `LCD_LayerInit`, `Colour_Set`, `framebuffer_init`) are kept unconditionally. `USE_GDEMWIN` is defined in Keil project settings (Cads → VariousControls → Define).

### 3. LCD_Init renamed to BSP_LCD_Init

To avoid conflict with emWin's `LCD_Init()`.
**Changed in**: `User/lcd/bsp_lcd.c:174`, `User/lcd/bsp_lcd.h:276`

### 4. gd32h7xx_it.c: SVC/PendSV/SysTick removed

FreeRTOS provides these handlers via `port.c` + `FreeRTOSConfig.h` macros. The original `gd32h7xx_it.c`'s `SysTick_Handler` (which called `Delay_Decrement`) is removed. When using FreeRTOS, do NOT call `Systick_Int()` — FreeRTOS's `vPortSetupTimerInterrupt` configures SysTick during `vTaskStartScheduler()`.

### 5. MPU configuration ORDER matters

MPU must be configured **BEFORE** enabling D-Cache. If cache is enabled first, stale cache lines with wrong attributes may be allocated for the SDRAM region, causing TLI to read garbage.

### 6. GUI_X_FreeRTOS semaphore initial count

Must use `xSemaphoreCreateCounting(1, 1)` — initial count 1 (one available token). Using count 0 causes the first `GUI_X_Lock()` to block forever.

### 7. Project file (.uvprojx) is based on TLI_LCD_5inches

The working .uvprojx was copied from `TLI_LCD_5inches/Project/Fire_H759IM_FSMC.uvprojx` and modified. Key settings:
- `RegisterFile` must be set (not empty)
- `Flash2` must be `BIN\UL2CM3.DLL` (not `CMSIS_AGDI.dll`)
- No `GD32H7XX` define needed (auto-defined in gd32h7xx.h)
- Define: `USE_GDEMWIN` (for bsp_lcd.c guards)
- Scatter file: references non-existent `stm32-ram.sct` — Keil auto-generates from device memory map

## AppWizard Integration Status

**INTEGRATED.** The AppWizard project is at `AppWizard/`.

**How it works**:
- `GUITask` calls `GUI_Init()` then `MainTask()` (defined in `AppWizard/Source/APPW_MainTask.c`)
- `MainTask()` calls `WM_SetDesktopColor()`, `APPW_X_Setup()`, `APPW_Init()`, manually creates XBF fonts via `_InitFont()`, then `APPW_CreateRoot()` + main loop
- Screen resources: `AppWizard/Source/Generated/` (Resource.c/h, ID_SCREEN_00.c/h, APPWConf.c)
- Fonts: `AppWizard/Resource/Font/` (compiled-in C arrays)
- Text: `AppWizard/Resource/Text/` (compiled-in C arrays)

### Critical AppWizard Configuration

1. **GUI_USE_ARGB=1** — REQUIRED. The precompiled `GDemWin_CM7_OS_DP.lib` was built with ARGB byte order. If set to 0 (ABGR), AppWizard widget colors are wrong (red/blue swapped, many colors invisible).

2. **GUI_SUPPORT_MEMDEV=1** — REQUIRED. Without it, AppWizard.h:74 preprocessor guard (`#if GUI_WINSUPPORT && GUI_SUPPORT_MEMDEV && ...`) excludes all AppWizard types, causing build errors.

3. **Manual XBF font creation** — REQUIRED in `APPW_MainTask.c: _InitFont()`. The AppWizard library's internal font creation fails with some XBF font sub-formats (GUIX-magic fonts). Pre-create fonts by calling `GUI_XBF_CreateFont()` with `GUI_XBF_TYPE_PROP_AA4_EXT` and store in `APPW__aFont[n]` before `APPW_CreateRoot()`.

4. **_MultibufEnable=0** — REQUIRED. Embedded single-buffer TLI (GUIDRV_LIN_16) does not support multi-buffering.

5. **APPW_PROJECT_PATH="/"** — Use a non-empty path. Empty string may cause `APPW_Init()` to misbehave.

6. **GUITask stack ≧ 4096 words** — AppWizard screen creation needs more stack than bare emWin.

7. **emWin pool ≧ 128KB** — AppWizard + XBF fonts need more heap than bare emWin's 64KB.

### AppWizard Workflow for Adding New Screens

1. Design in AppWizard tool → export to `AppWizard/Source/Generated/`
2. Add new `.c` files to Keil AppWizardSource group
3. If new fonts: add `.c` files to AppWizardResource group, add `_InitFont()` call in `APPW_MainTask.c`
4. If new bitmaps: export as C arrays, add to AppWizardResource group
